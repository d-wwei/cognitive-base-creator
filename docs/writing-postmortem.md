# 写作复盘：认知底座文章（article-zh v1→v3）

本文档记录"认知底座"文章从初稿到定稿的完整修改过程，包括遇到的问题、用户反馈、以及应被 great-writer 技能吸收的改进建议。

---

## 背景

### 文章目标
为 GitHub 仓库 `d-wwei/cognitive-base-creator` 写一篇中文推广文章，介绍"认知底座"（Cognitive Base）这个概念——一种改变 AI Agent 思维方式（而非能力范围）的元认知指令集。目标读者是已经在使用 AI Agent + Skill 的技术用户。

### 核心概念
- **认知底座**：写入 Agent 宪法级配置文件（如 CLAUDE.md、AGENTS.md）的元认知规则，跨对话跨项目始终生效
- **与 Skill 的区别**：Skill 是被调用的工具（触发→加载→执行→卸载）；认知底座是 Agent 的身份定义（始终在线）
- **双层结构**：30 行核心规则写入宪法配置 + 120 行完整框架作为 Skill 按需参考
- **轴线独立**：每个底座改变一个思维轴线（分析方式、完成标准、判断力），互不干扰

### 策略约束
用户有 20 个认知底座，但决定文章中**只公开 3 个**（第一性原理、以终为始、Tacit Knowledge），其余留作后续逐步发布的内容储备。这个决策是在 v2 写完后做出的。

### 版本演进
1. **v1**（article-zh.md）：初版，介绍概念 + 3 个场景 + 设计原则。问题：过于全面但不够锋利
2. **v2**（article-zh-v2.md）：基于 20 个底座的完整生态介绍。被否决——暴露太多内容
3. **v3**（article-zh-v3.md）：战略性重写，只露 3 个底座，经历 3 轮详细修改后定稿

## 修改轮次与问题清单

### 第一轮反馈（7 个问题）

#### 问题 1：开头太长，3 个例子稀释了冲击力
**初稿做法**：开头用了 3 个并列的场景（MBA 决策、项目方案跳过风险评估、调查问题不验证）来展示 Agent 的默认缺陷。

**用户反馈**："开头的 3 个例子感觉有点长"

**问题本质**：3 个例子是并列关系，每个都在说同一件事——"Agent 的默认思维方式有缺陷"。第一个例子（MBA）已经足够有力，后两个是冗余佐证，不是递进。读者读到第二个例子时已经理解了论点，第三个例子开始产生"好了我懂了"的阅读疲劳。

**修改方向**：保留 MBA 作为主钩子（最有共鸣的场景），后两个压缩为一两句话的补充。

**great-writer 应吸收的规则**：
> Hook 部分的多个例子必须是递进关系（每个例子加深理解或揭示新维度），不能是并列关系（都在说同一个论点）。如果 3 个例子在论证同一件事，砍到 1 个 + 极简补充。读者的注意力预算最珍贵的是在开头——不要用并列例子消耗它。

---

#### 问题 2：Skill 局限性部分缺少关键论点
**初稿做法**：讲了 Skill 的两个局限——覆盖不到所有场景 + 需要手动触发。

**用户反馈**："第二部分讲 Skill 的局限，需要补充一个关键点：即使 Skill 有智能触发，你也不知道它什么时候在生效、什么时候没生效。"

**问题本质**：初稿把 Skill 的智能调用当作一个已解决的问题来处理（"不需要手动输入，Agent 会自动判断"），但实际上这恰恰暴露了更深的问题——自动调用意味着你**无法确认**此刻 Skill 是否在生效。Agent 的表现变成黑盒：时好时坏。这比"需要手动触发"更致命，因为它破坏了用户对 Agent 能力的可预期性。

**修改方向**：把"你不知道它什么时候在生效"作为 Skill 局限的第一个论点（比"覆盖不到所有场景"更有冲击力）。

**great-writer 应吸收的规则**：
> 当论证"X 的局限"时，不要只从 X 做不到的角度切入。要同时审视 X 声称做到了的部分——那里往往藏着更深的问题。"它能自动触发"表面上是优点，但"你无法确认它有没有触发"是从同一个特性推导出的致命缺陷。好的局限性论证要能把对方的卖点翻转成弱点。

---

#### 问题 3：MBA 例子在文章中重复出现
**初稿做法**：MBA 出现在开头的 hook 里，然后在"三个场景"部分又作为场景一完整展开。

**用户反馈**："MBA 的例子感觉很重复"

**问题本质**：同一个例子在两个不同的结构位置承担了不同的叙事功能（hook vs 场景对比），但读者感知到的是"又是 MBA"。写作者关注的是结构完整性（hook 需要一个例子、场景对比需要三个场景），但读者关注的是信息新鲜度。

**修改方向**：MBA 只在开头用一次（作为 hook），场景对比部分的场景一直接复用开头已建立的上下文，不再完整复述。

**great-writer 应吸收的规则**：
> 同一个例子在文章中只能完整展开一次。如果一个例子在 hook 部分已经讲过，后面引用时应该用"回到开头那个 MBA 的例子"一句话带过，然后直接进入新增信息（装了底座后的对比）。不要因为结构需要而重复内容——结构服务于阅读体验，不是反过来。跟踪每个例子的首次出现位置，避免让读者产生"这个例子我已经看过了"的感觉。

---

#### 问题 4：产品名称过时
**初稿做法**：把其中一个底座叫"what is good job"。

**用户反馈**："what is good job 已经改成'结果导向，以终为始'了，仓库名是 d-wwei/results-driven"

**问题本质**：写作时使用了代码/开发阶段的内部名称，而非面向用户的产品名称。

**great-writer 应吸收的规则**：
> 当文章涉及具体产品/仓库名称时，必须在写作前确认最新的公开名称和仓库 URL。不要假设开发名 = 发布名。如果用户提供了仓库链接，读取仓库的 README 确认当前名称。

---

#### 问题 5：对 Tacit Knowledge 的定性不准确
**初稿做法**：把 Tacit Knowledge 描述为改变"Agent 怎么组织和表达它的思考"。

**用户反馈**："Tacit Knowledge 改变的只是 agent 怎么组织和表达它的思考吗？"

**问题本质**：把一个改变**判断力本身**的底座降格成了一个改变**表达方式**的工具。Tacit Knowledge 的核心不是"说话更像专家"，而是"思考更像专家"——它改变的是 Agent 注意什么、跳过什么、浮现什么隐性信号。这些是判断力的构成，不是修辞技巧。

**修改方向**：重新定位为"改变判断力本身"——信息密度质变（从"正确但你已经知道的"到"你不知道自己需要知道的"），而不只是表达风格变化。

**great-writer 应吸收的规则**：
> 描述一个产品/工具的核心价值时，区分"改变了什么"和"看起来改变了什么"。用户能观察到的表面变化（说话方式）不等于底层改变（判断力）。好的产品描述应该从底层改变出发，让表面变化成为自然推论。如果你发现自己在用"组织和表达"来描述一个号称改变"思维方式"的产品，检查是否降格了它的定位。

---

#### 问题 6：文章存在自相矛盾
**初稿做法**：在设计原则部分写了"零理论术语名称"的约束——认知底座的规则不应该包含学术术语。

**用户反馈**：文章本身就充满了理论术语（第一性原理、元认知、贝叶斯推理），自相矛盾。

**问题本质**：把认知底座的内部设计原则（指令文本中不用术语）错误地呈现为面向用户的约束。这是混淆了"产品的实现细节"和"面向读者的论述"。

**修改方向**：删除这个约束小节。它是开发者视角的设计笔记，不是读者需要知道的内容。

**great-writer 应吸收的规则**：
> 写完每个小节后做一次一致性检查：这个小节的论点和文章其他部分有没有矛盾？特别注意"约束类"和"原则类"的陈述——如果你说"不用 X"，确保文章自身没有在用 X。另外，区分"产品的内部设计决策"和"读者需要知道的信息"。除非内部设计决策直接影响用户理解或使用，否则不要放进面向用户的文章。

---

#### 问题 7：缺少关键机制解释
**初稿做法**：说了认知底座"始终生效""跨对话跨项目"，但没解释**怎么做到的**。

**用户反馈**："整篇文章还是没说清楚认知底座是怎么跨对话跨项目随时发挥作用的"

**问题本质**：文章反复声称一个 capability（始终在线），但从未解释实现 mechanism（写入宪法级配置文件）。对技术读者来说，"始终在线"如果没有 mechanism 支撑，就是一个未经验证的营销宣言。他们会问"怎么做到的？"，如果文章不回答，就失去了可信度。

**修改方向**：在"什么是认知底座"部分明确说出：写入 `CLAUDE.md` / `AGENTS.md` / `system instruction`，这些是 Agent 启动时最先读取的身份定义文件。这就是"始终在线"的 mechanism。

**great-writer 应吸收的规则**：
> 当文章声称一个产品具有某种能力（特别是听起来不寻常的能力，如"跨对话始终生效"）时，**必须**在声称后的 2-3 段内解释实现机制。技术读者对"是什么"的兴趣远小于对"怎么做到的"的兴趣。pattern: 声称 → 机制 → 所以（推论）。如果只有声称没有机制，读者会把它归类为营销话术而非技术事实。

### 第二轮反馈（5 个问题）

#### 问题 8：三个场景的叙事重心错位
**初稿做法**：三个场景对比（人生决策、商业判断、内容创作）的叙事重心放在"认知底座始终在线"——强调的是"这三个场景没有人调用任何 Skill"。

**用户反馈**："'同一套规则，三个完全不同的场景'里我们要强调的不是认知底座始终在线，而是思考方式的改变"

**问题本质**：写作者在用场景展示 mechanism（始终在线，无需调用），但读者需要看到的是 outcome（思考方式变了→回答质量变了）。"没有人调用 Skill 但它自动生效了"是开发者关心的特性；"Agent 面对完全不同的领域都在做同一件事：先审计假设"才是用户能感知的价值变化。

**修改方向**：每个场景的对比要聚焦在 Agent 思考行为的变化（默认接受框架 vs 先审计假设），而不是调用机制的变化（手动触发 vs 自动生效）。

**great-writer 应吸收的规则**：
> 场景对比的叙事重心必须是用户可感知的行为变化，不是产品的技术机制。用户不关心"它是怎么自动生效的"，用户关心"它生效后我看到了什么不同"。检查方法：把每个场景的总结句拿出来——如果它在说 mechanism（"不需要调用""自动匹配"），改成说 outcome（"面对任何问题都先审计假设""思维方式本身变了"）。

---

#### 问题 9："硬约束"部分不应存在
**初稿做法**：有一个独立的"两个硬约束"小节，讲认知底座的设计原则——轴线独立和零理论术语。

**用户反馈**："两个硬约束部分是不是干脆去掉算了？"

**问题本质**：这些是产品设计的内部约束，不是读者需要知道的信息。它们对理解和使用认知底座没有帮助，反而打断了从"这是什么→怎么用→去哪装"的读者路径。

**great-writer 应吸收的规则**：
> 产品介绍文章中，删除所有"设计者视角"的内部约束/原则，除非它们直接影响用户的使用决策。检查标准：如果删掉这个小节，用户能不能正常理解和使用产品？如果能，删掉。保留的内容应该服务于读者路径：问题→方案→机制→证据→上手。

---

#### 问题 10：v2 的全量暴露策略错误
**v2 做法**：把全部 20 个认知底座都列出来，详细介绍每个轴线。

**用户反馈**："我有点纠结，这样相当于把我全部的 20 个认知底座都公布出去了... 要不要分批一个一个公布，会更好呢？"

**问题本质**：写作者默认了"信息越多越好"，没有考虑内容发布的战略维度。20 个底座一次性公布 = 一次性消耗了所有内容弹药。分批发布 = 每个底座都是一次内容事件，持续产生关注。

**great-writer 应吸收的规则**：
> 当文章涉及一个产品系列或多个组件时，写作前要确认发布策略：一次性全量 vs 分批发布。不要自动假设"完整介绍 = 好文章"。如果用户有 N 个产品/组件，问一句："全部放进来还是选代表性的几个？" 内容策略先于写作执行。

---

### 第三轮反馈（补充问题）

#### 问题 11：认知底座 vs Skill 的区别需要更清晰的结构化对比
**反复出现的问题**：在多轮修改中，用户反复强调"认知底座和普通 Skill 的不同可能还要写清楚"。初稿倾向于用散文式的描述来解释区别，但读者需要的是清晰的结构化对比。

**最终方案**：分两个维度对比（生效机制 + 指令层面），每个维度都有配图，分别用流程图和层级图来可视化。

**great-writer 应吸收的规则**：
> 当文章的核心论点是"A 和 B 的区别"时，散文式描述不够——需要结构化对比。至少提供一个维度的表格/图表对比。如果区别有多个维度，每个维度单独展开。读者应该在 10 秒内能回答"A 和 B 到底哪里不同"，如果需要读完两段话才能提取出区别，说明结构化不够。

## 总结：great-writer 应吸收的改进方向

### A. 结构层面

| # | 规则 | 触发条件 |
|---|------|---------|
| 1 | Hook 的多个例子必须递进，不能并列 | 开头有 2+ 个例子 |
| 2 | 同一个例子只完整展开一次，后续引用用一句话回指 | 例子在文章中出现 2+ 次 |
| 3 | 删除"设计者视角"的内部约束/原则小节 | 存在不影响用户理解的设计笔记 |
| 4 | 写作前确认内容发布策略（全量 vs 选择性） | 文章涉及产品系列/多组件 |

### B. 论证层面

| # | 规则 | 触发条件 |
|---|------|---------|
| 5 | 把对方卖点翻转成弱点，而不只列做不到的事 | 论证"X 的局限" |
| 6 | 声称能力后 2-3 段内必须解释实现机制 | 声称了不寻常的能力 |
| 7 | 场景对比聚焦用户可感知的行为变化，不是技术机制 | before/after 对比场景 |
| 8 | 每个小节写完后做一致性检查（有无自相矛盾） | 存在"约束类""原则类"陈述 |

### C. 定性层面

| # | 规则 | 触发条件 |
|---|------|---------|
| 9 | 区分"底层改变"和"表面改变"，从底层出发描述 | 描述产品核心价值 |
| 10 | 确认产品/仓库的最新公开名称 | 文章包含产品名和链接 |
| 11 | "A vs B 的区别"需要结构化对比，不能只用散文 | 核心论点是两个概念的区别 |

### D. 元规则

**最重要的一条**：区分三种视角，不要混用——
- **开发者视角**（内部设计决策、技术约束、实现细节）→ 只在必要时简要提及
- **产品视角**（能做什么、怎么做到的、和替代方案的区别）→ 文章主体
- **用户视角**（我装了之后会看到什么变化、怎么装、装完怎么验证）→ 最重要，必须能从文章中直接提取出行动

初稿中大量问题的根源都是：写着写着滑进了开发者视角（设计约束、轴线独立性、零术语原则），而读者需要的是产品视角和用户视角。
