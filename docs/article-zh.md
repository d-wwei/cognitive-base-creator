# 所有 Agent 都缺了 Skill 无法解决的一层：认知基座

**TL;DR：** 给 Agent 装再多 Skill 也教不会它思考。真正需要的是改变它的全局思维模式——这就是认知基座做的事：不教它做更多事，而是改变它面对所有任务时怎么想。

---

你给 Agent 装了写作 Skill、分析 Skill、设计 Skill。它能帮你写周报，能帮你做竞品分析，能帮你出设计稿。

然后你问它："我该不该去读 MBA？"

> MBA 有几个主要优势：1. 校友网络 2. 商业知识体系 3. 职业转型跳板。同时也有劣势：1. 时间成本（2 年）2. 学费高昂 3. 机会成本。是否值得取决于你的职业目标、财务状况和所在行业。建议和校友聊聊，做更深入的评估。

一堆正确但无用的话。你读完还是不知道答案。

你让它帮你写一份完整的项目方案——要包含背景分析、方案设计和风险评估三部分。它写了背景和方案，跳过风险评估，说"方案已完成！"

你让它帮你查一个问题的原因。它看了一眼，说"可能是配置问题，建议你自己检查一下。"——但它自己什么都没查。

Skill 装了 100 个，问题还是同一类。

因为所有 Skill 都在解决"做什么"。没有 Skill 在解决"怎么想"。

---

## 不是 Skill 的问题，是缺了一层

你可能会说：那装一个"更好的分析 Skill"或者"更严格的输出规范"不就行了？

市面上确实有这类工具——各种 harness、system prompt 模板、agent 行为约束仓库。思路是：写更细的规则，管住 Agent 的输出质量。

但它们有一个共同的局限：**绑定任务类型**。

一个"更好的分析 Skill"能让 Agent 做分析时不列利弊，但管不到它写方案时跳过难的部分。一个"更严格的代码审查规则"能让 Agent 写代码时更谨慎，但管不到它给你人生建议时还在和稀泥。

你需要的不是更多 Skill。你需要一个**跨所有任务都生效的思维层**。

用同一套 30 行规则，在三个毫不相关的场景里看看效果。

## 同一套规则，三个完全不同的场景

以下三个对比，全部来自同一个认知基座——[第一性原理](https://github.com/d-wwei/first-principles)。装一次，永远在线。

### 场景一：人生决策

**问**："我该不该去读 MBA？"

没装认知基座：
> MBA 有几个优势：校友网络、商业知识、职业跳板。劣势：2 年时间、高学费、机会成本。是否值得取决于你的职业目标和财务状况。建议和校友聊聊。

装了之后：
> 先别问"该不该读"，先拆开看你到底想买什么。MBA 本质上是四样东西捆绑销售：学历信号、人脉圈子、系统知识、职业暂停键。
>
> 逐个检验：学历信号——只在金融、咨询和大公司晋升中是硬通货，翻一下你目标岗位的招聘要求，有多少写了 MBA Required？人脉——YC 社区、付费圈子也能提供，成本低一个数量级。系统知识——线上商学院课程接近免费。职业暂停键——请半年假也能实现。
>
> 真正的问题是：这四样里，哪一样是你目前的瓶颈？如果是学历信号且目标行业确实看这个，去读。如果是其他三样，有更便宜的替代方案。

### 场景二：商业判断

**问**："公司还有 18 个月的钱，投资人现在有兴趣，该不该融资？"

没装认知基座：
> 融资有几个考虑因素：1. 市场时机 2. 估值预期 3. 稀释比例 4. 资金用途。一般建议是"从强势位置融资"，趁投资人有兴趣时把握机会。可以先和几家聊聊，了解条件。

装了之后：
> "该不该融资"里藏着一个假设——融资时机是"要么抓住要么错过"的窗口。这是行业惯例，不是事实。
>
> 回到基本面：你的烧钱速度是多少？下一个关键里程碑要花多少钱？核心指标是逐月变好还是走平？
>
> 如果指标逐月改善，等一等 = 更高估值、更少稀释。如果走平或不确定，资本的期权价值高于稀释成本——现在拿。
>
> "从强势位置融资"假设强势是单调递增的。但如果你的增长有拐点，等待可能让你错过窗口。真正的决策分支：你的轨迹在加速还是在减速？

### 场景三：内容创作

**问**："我在写一本关于 AI 的书，写到中间感觉散了。每章单独看没问题，但连起来没有推进感。"

没装认知基座——你会拿到一个六条建议的清单：梳理主题线、用叙事弧串联、每章结尾设悬念、加贯穿全书的案例、请人试读、先写完再改结构。正确，但全是你自己也能想到的。

装了之后：
> 你的问题不是"章节之间缺少连接"，而是你的章节是**并联电路**——每章独立供电，读者的注意力在分散而非累积。你需要的是**串联**——每章的输出喂给下一章，能量逐级叠加。
>
> 检验一下：你第一章提出了一个读者真正想知道答案的问题吗？如果有，后面的每一章是在加深这个问题的紧迫感，还是在讲一个新话题？
>
> 交响乐的推进感不是因为每个乐章都好听，而是因为它制造了未解决的和声张力——听众被驱动着想听到解决。你的书需要一个"未解决的不协和音"——第一章提出，最后一章才完全回答，中间每章要么加深紧迫感，要么给出部分回答同时引出更深的问题。
>
> 如果某一章既不加深问题也不回答问题，它在中断推进感，不管它自己多好。

---

三个场景——人生决策、商业判断、内容创作——没有任何领域重叠。但 Agent 在每个场景里做了同一件事：不接受问题的原始框架，先审计里面的假设，然后从基本面重建。

一个分析 Skill 能改善场景一和场景二，但对场景三无能为力。一个写作 Skill 能改善场景三，但对前两个没帮助。

这三个场景靠的是同一套 30 行规则。这就是认知基座在做的事。

---

## 什么是认知基座

认知基座（Cognitive Base）是一类特殊的 AI Agent 指令集。文件格式上和普通 Skill 没有任何区别——都是 Markdown 里的自然语言指令。区别在于指令作用的层面。

普通 Skill 的指令是**操作层**的："使用 8px 网格系统""API 响应格式统一为 JSON""标题不超过 20 字"。绑定特定任务，离开那个任务就没用了。

认知基座的指令是**元认知层**的："回答之前，先分辨哪些是事实、哪些是惯例""每个完成声明都需要附带证据""先诊断整体，再修局部"。不管 Agent 在做什么任务，这些思维习惯都在起作用。

类比：普通 Skill 是菜谱，认知基座是刀工。一百本菜谱让厨师能做更多的菜，但刀工不行，做什么都差那么一点。

这不是一个凭空发明的概念。学术界其实已经在讨论同一件事——Nature 今年的 [Fast, slow, and metacognitive thinking in AI](https://www.nature.com/articles/s44387-025-00027-5) 指出 AI 需要一个元认知层来监控和调节自己的思维过程；Microsoft 的 [元认知教程系列](https://techcommunity.microsoft.com/blog/educatordeveloperblog/ai-agents-metacognition-for-self-aware-intelligence---part-9/4402253) 把这个方向定义为"thinking about thinking"；Google Cloud [写了专门的设计指南](https://medium.com/google-cloud/designing-meta-cognitive-agent-skills-ce9db69821d9)；[剑桥 ICML 论文](https://news.qq.com/rain/a/20250618A09HIP00)甚至证明了具备元认知能力的 Agent 在自主学习上显著优于没有的。方向是公认的。但到目前为止，这些工作全部停留在论文和概念层面——没有一个可安装的、30 秒能跑起来的产品。

### 和 harness / prompt 优化工具的区别

![领域 Skill vs Harness 工具 vs 认知基座](images/compare.png)

Harness 工具的思路是"管住输出"——更严格的规则、更详细的检查清单。有价值，但本质还是操作层：规定了在特定场景下怎么做。认知基座改变的是推理起点——不规定在哪个场景做什么，而是改变 Agent 面对任何问题时的思考方式。这就是为什么 MBA 决策、融资判断、写书结构这三个毫不相关的场景，能被同一套规则同时改善。

### 设计上的几个关键点

认知基座始终在线，没有触发条件。实现方式是把核心规则（约 30 行）注入到 Agent 的宪法级配置文件——Claude Code 的 `CLAUDE.md`、Codex 的 `AGENTS.md`、Gemini 的 `system instruction`。这些文件每次对话都加载，不需要用户说"请用第一性原理分析"。

架构上分两层：30 行核心规则是宪法——精炼到不能再精炼的元认知指令，始终生效；120 行完整框架是实施细则，作为 Skill 文件按需参考，包含认知转换的详细说明、反模式清单、示例、和其他基座的组合接口。

![双层架构：核心规则 + 完整框架](images/arch.png)

平台无关。认知基座本质上是一段文本，任何能读取启动指令的 Agent 都能用——Claude Code、Codex、Gemini、Cursor、Cline、OpenCode，不挑。

天然可叠加。不同的认知基座作用在不同轴线上，互不干扰。和领域 Skill 也不冲突——认知基座决定思考顺序，领域 Skill 决定具体参数。操作系统和应用程序的关系。

## 不止一个基座：不同轴线，不同能力

前面用第一性原理展示了"改变分析方式"这根轴线。但 Agent 的思维缺陷不止分析这一种。

### What Is Good Job：改变完成标准

Agent 还有一个顽疾：分不清"我做了事"和"目标达成了"。你让它做三件事，它做了两件就说"搞定了"。它调查一个问题，看了一眼就说"可能是 XX 原因"——没有任何验证。

[What Is Good Job](https://github.com/d-wwei/what-is-good-job) 也是 30 行核心规则，但作用在完全不同的轴线上——不改变 Agent 怎么分析，而是改变它怎么判断"完没完"：

| 默认模式 | 目标模式 |
|---|---|
| 做了事 = 完成 | **结果**——用户目标达成 = 完成 |
| "搞定了"基于自信 | **证据**——"搞定了，证据在这" |
| 试一次就放弃 | **穷尽**——3 种以上根本不同的方法 |
| 做了主要的就行 | **完整**——重读需求，交付每一部分 |
| 不确定就问用户 | **主动**——先自己查，只问真查不到的 |

同样跨所有任务生效——不管是写方案、做调研还是排错。

### 叠加效果

- **第一性原理** 管 Agent 怎么**分析问题**——审计假设、拆到基本面、自由重构
- **What Is Good Job** 管 Agent 怎么**评估自身工作**——结果导向、有证据、不遗漏

只有第一性原理 = 精彩的分析但做了一半。只有 What Is Good Job = 彻底完成了平庸的思考。两个都装 = 想得透彻，干得干净。

不冲突，因为管的是不同的事。就像操作系统同时有文件系统和内存管理——各管各的。

## 如何制作一个认知基座

认知基座不是什么神秘技术。它就是一个 Markdown 文件，里面是元认知层的操作指令。但"元认知"和"操作"之间的边界，是制作过程中最容易踩的坑。

### 核心区分

✅ "回答之前，先分辨哪些是事实，哪些是惯例" — 元认知，改变推理起点，任何任务都适用

❌ "使用决策矩阵打分" — 操作，规定具体工具，只在分析场景适用

测试方法：这条指令在写方案、做分析、给人生建议时都能用吗？如果只在某个场景适用，它是操作层的，不该出现在认知基座里。

### 三个硬约束

**零理论名字。** 核心规则文件里不许出现"贝叶斯""第一性原理""默会知识"这些词。纯操作指令。Agent 不需要知道理论来源，只需要执行规则。去掉标题后还能猜出是什么理论？太学术化了，继续精炼。

**每条指令可直接执行。** 禁用模糊动词——"understand"、"consider"、"be aware of"不是动作。好的动词是"identify"、"list"、"check"、"reverse"、"compare"、"state"。

**改变推理过程，不规定输出格式。** 不说"用表格呈现"，而说"先确定焦点再安排辅助信息"。前者绑死格式，后者改变思维顺序。

### 生产工具

我们把这个制作流程标准化了——[Cognitive Base Creator](https://github.com/d-wwei/cognitive-base-creator) 是一个 Skill，给它任意一个思维框架名称（"贝叶斯推理""系统思维""逆向思维"），它输出完整的可安装文件包：30 行核心规则、120 行完整框架、反模式清单、Before/After 示例、多平台安装指南。它本身也是平台无关的，在任何 Agent 上都能跑。

## 往哪走

目前的 Agent Skill 生态只有一层——领域 Skill。所有人都在做"让 Agent 能做更多事"。

如果认知基座成为标配，会多出一层——**认知层**。不同的基座编码不同的思维风格，你可以选择适合自己工作方式的基座组合，而不是为每种任务都找一个专门的 Skill。

不过这是远景。眼前更重要的是：认知基座能不能跨模型、跨平台稳定地改善 Agent 的推理质量？这需要更多实践检验，也需要更多人参与进来——用自己的思维框架做基座，在自己的工作流中测试效果。
